# Architecture Notes

Working notes on structure and known tradeoffs, so decisions don't get lost between sessions. Not a spec — see `twin-star-spec.md` for design intent. This is about how the code is organized.

## Current layout

```
src/game/
  vec2.ts        vector math + rotation helpers
  poly.ts        polygon geometry (clipping, point-in-polygon, boundary queries)
  physics.ts      shared 2D rigid-body impulse resolver (mass + moment of inertia + torque)
  constants.ts    every tunable value, flat, one file
  input.ts        keyboard/mouse state
  ship.ts         Ship class (movement, resources)
  asteroid.ts     Asteroid + Cell + composition data, Voronoi generation, adjacency graph
  chunk.ts        collectible debris entity
  tools.ts        tool definitions (laser/drill/charges)
  engine.ts       Engine class — simulation only: update loop, physics, mining, input handling
  renderer.ts     Renderer class — draws one frame from Engine's state, owns nothing
src/components/
  GameCanvas.tsx  mounts canvas, owns the rAF loop, wires Engine + Renderer together
```

## Done: engine/renderer split (this session)

`engine.ts` had grown to 1222 lines — simulation, mining logic, *and* every render call in one class. Split rendering out into `renderer.ts`. `Engine` now only does simulation (785 lines); `Renderer` only reads Engine's state and draws (459 lines). `Renderer` takes `engine: Engine` as a parameter rather than owning any state itself.

This was done now specifically *because* enemies/combat are next on the build order (spec §13 step 5) and would have hit this file first. The render side already follows a one-method-per-entity-type pattern (`renderShip`, `renderAsteroid`, `renderChunk`) — adding `renderEnemy` when the time comes is a drop-in, not a restructure. Same on the simulation side: an `enemies: Enemy[]` array and an `updateEnemies(dt)` step slot into Engine's existing update loop shape.

To make this work, a few things that were `private` on `Engine` became public surface for `Renderer` to read: `screenToWorld()`, `currentTarget`, `blastEffects`. Nothing else changed — `Engine` still fully owns mutation of its own state; `Renderer` only reads.

## Done: decoupling mechanics from their one current trigger (this session)

A recurring smell: a mechanic implemented only inside the one call site that currently needs it, rather than as a capability anything could reach. `Explosion`/`Engine.applyExplosion()` was the first fix (charges used to trigger blast logic directly; now charges just raise a generic `Explosion`). Same pass applied to two more:

- **Tool signature cost** used to be a constant hand-matched to each tool's own code (`ship.addSignature(LASER_SIG_PER_SEC * dt)` inside `cutCell`, etc.) — nothing enforced it stayed in sync with the tool itself. Moved onto `ToolDef` (`sigPerSecond` / `sigPerUse` in `tools.ts`), read generically at each call site. A new tool now declares its own cost instead of needing a new hand-wired call.
- **Collision impact damage** was inline math only `resolveAsteroidCollision` knew how to do. Pulled into `Engine.applyCollisionImpact(closingSpeed, source)` so a future enemy-ramming collision resolver can call the same formula instead of reimplementing it.
- **`Ship.takeImpact`** now requires a `DamageSource` (`"collision" | "explosion"`) on every call, mirroring `Explosion.source` — not consumed anywhere yet, but future per-source feedback (distinct hit messages/sounds) has something to switch on without a retrofit across call sites.

Left alone on purpose: `Cell.hasCharge` is charge-specific boolean state on a cell. It looks like the same smell, but there's only ever been one thing that attaches to a cell — generalizing to a device-type enum now would be designing for a feature that doesn't exist. Rule of three still applies.

Also generalized: cursor highlighting used to be tool-specific (`currentTarget`, range/mode-gated, drove both actual mining *and* the visual outline). Split into two concepts — `Engine.currentTarget` still drives mining eligibility, but `Engine.hoverTarget`/`hover.ts` is a new, independent "what's under the cursor" query that runs every frame regardless of tool or mode, and covers chunks as well as cells. The white outline + tooltip moved to their own render pass (`Renderer.renderHoverHighlight`) drawn after everything else, which as a side effect fixed a z-order bug (drill fracture lines used to render on top of the old per-cell highlight stroke).

## Done: one mass system for every collision (this session)

Found while auditing the above pattern: the ship had *two* masses. `SHIP_MASS = 30` was the real constant, used in ship-vs-rock — but `resolveChunkShipBumps` independently invented `SHIP_RADIUS * SHIP_RADIUS * 4` (324, a 10x disagreement) rather than using it. Chunk mass was `radius * radius` with no density constant at all, calibrated against nothing. Three unrelated mass conventions for what should be one physical model — almost certainly the concrete cause of the "big things push too easily, size/mass doesn't matter" note in the spec.

Fixed by making mass a single well-defined property per body type: `SHIP_MASS` for the ship, and `Chunk.mass` (a getter sharing `ROCK_MASS_PER_AREA` with rock cells — a chunk is the same material, just detached, computed from its circular radius). A chunk is now properly much lighter than the cell it came from, on the same scale rock mass already used. This mass model was then folded into the generic collision system below, superseding the short-lived `rigidRefForShip`/`rigidRefForChunk` helpers from earlier in the session.

## Done: one generic collision system instead of one method per pair of kinds (this session)

Prompted by a direct question: mass/velocity were already unified (`RigidRef`/`resolveContact`), so why were there still five hand-written collision methods (`resolveAsteroidCollision`, `resolveChunkCollisions`, `resolveChunkChunkCollisions`, `resolveChunkShipBumps`, plus rock-rock) each re-encoding which concrete lists collide with which? What happens when enemies, drones, or satellites need to be added — a sixth, seventh, eighth method, each a variation on the same math? That's the same "trigger vs capability" smell as `Explosion`, just in the physics layer.

Two things were genuinely hardcoded per pair rather than derived:
- **Restitution/friction were hand-tuned constants per pairing** (`COLLISION_RESTITUTION`, `CHUNK_CHUNK_RESTITUTION`, `CHUNK_SHIP_BUMP_RESTITUTION`, `CHUNK_ROCK_RESTITUTION`, `ROCK_ROCK_RESTITUTION` — five numbers for three actual materials). Replaced with `Material` (`physics.ts`: `{ restitution, friction }`) — one per body kind (`SHIP_MATERIAL`, `ROCK_MATERIAL`, `CHUNK_MATERIAL` in `engine.ts`) — and `combineMaterials(a, b)` derives any pairing's behavior (restitution averages, friction multiplies) instead of needing a constant hand-tuned for every existing kind a new one might touch. Friction in particular used to exist only as an unnamed `* 0.92` scrubbed onto the ship post-collision, nowhere else — chunks skated frictionlessly off everything. It's now a first-class part of every material (`applyFriction` in physics.ts), so chunks have real (light) grip too.
- **The pairing itself was five separate broad-phase loops.** Replaced with `Body`/`BodyHandle` (new `body.ts`) — any circle-shaped, independently-moving thing (today: the ship and every chunk, via `Engine.circleBodies()`) collides with every other circle body (`resolveBodyPair`/`resolveCircleCollisions`) and with rock (`resolveBodyVsRock`) through one generic function each, driven by the body's own `mass`/`material`/`radius`. A future circular entity (a drone, a simple enemy) is one more line in `circleBodies()` — zero new collision code, automatically correct against the ship, every chunk, and rock alike.

Rock-vs-rock (`resolveGroupPair`) deliberately stays its own thing — it's genuinely different geometry (irregular polygon via drift groups, not a circle), not laziness. It already scales to "more rocks" for free, since a second asteroid's drift groups would just be more entries in the same `driftGroups` list.

Reactions that aren't pure physics (only a *solid hit against rock* damages the ship — a chunk bump doesn't, matching prior behavior exactly) live in one small `reactToCollision(kindA, kindB, closingSpeed)` dispatch, kept deliberately separate from the physics resolvers — extending "what reacts to what" is a one-line addition there, not a new resolver.

**Also fixed in the same pass, now that collision is unified:** tunneling at speed. Collision was only ever checked at each frame's *end* position, so something moving far enough in one frame (cruise top speed; a chunk ejected point-blank from an extraction) could skip clean over thin geometry — a freshly laser-cut sliver is easily thinner than one frame's travel. Fixed with substepping, not full continuous collision detection: `Engine.substepsFor(speed, dt)` splits a fast frame's ship-movement+rock-collision (and, separately, chunk-movement+rock-collision) into a few smaller steps, each with its own collision check. Free at normal speeds (returns 1 step), capped at `MAX_COLLISION_SUBSTEPS` so a runaway velocity can't blow up frame cost.

## Done: the hub — scene switch, materials (no currency), and a first upgrade (this session)

Built in two passes, deliberately minimal each time rather than the full spec §9 vision at once.

**Pass 1 — round-trip skeleton.** `Engine.scene: "field" | "hub"` — the field update short-circuits into a much simpler `updateHub` while docked (no physics, no mining, nothing at risk). `Hub` (new `hub.ts`) is a fixed point you fly to and dock at (press F in range, `Engine.updateDocking`) — no time pressure, no auto-return; docking is entirely the player's call. Cargo was flattened into a single number before this (`Ship.cargoUsed`); since materials need to be spent on real things, `Ship.cargo` is now a per-composition tally (`CargoHold = Record<Composition, number>`), and docking calls `ship.clearCargo()` straight into `hub.deposit()` — **no currency conversion**, materials sit in storage as themselves. The hub also registers as a radar `Contact`, but refreshed every frame rather than run through the normal ping/vision discovery gate — you always know where home is, unlike everything else on radar.

**Pass 2 — first upgrade, DOM overlay.** Adding an actual purchasable upgrade was the trigger the previous entry flagged for moving off hand-drawn canvas: a shop needs buttons/lists, not `ctx.fillText`. `HubOverlay.tsx` is a React component rendered by `GameCanvas.tsx` only while `engine.scene === "hub"`; canvas still owns the in-field HUD entirely, and now just paints a plain backdrop while docked. Since the game loop is otherwise fully imperative (Engine mutates itself, nothing re-renders React), `GameCanvas` bridges into React state by hand — diffing a small hub snapshot every frame and only calling `setState` when it actually changes, so docking doesn't turn into 60 wasted re-renders/sec. Overlay buttons call straight back into `Engine.purchaseUpgrade`/`Engine.launchFromHub` (both public); `Engine` stays the single source of truth. One real bug caught before shipping: `InputState` listens for `mousedown` on `window` for field input, so a click on an overlay button would otherwise leak through as a "ghost click" into mining input the instant a scene switch takes effect — fixed with `stopPropagation` at the overlay's root.

Upgrades themselves (`upgrades.ts`) are a small data table (`UPGRADES: Record<UpgradeId, UpgradeDef>`), same pattern as `COMPOSITION_INFO`/`TOOLS` — adding upgrade #2 is a new entry, not a new purchase method. Left deliberately un-generalized: *applying* an upgrade's effect is still hand-written in `Engine.purchaseUpgrade` (currently just `ship.cargoCapacity += bonus`). With only one upgrade there's nothing real to generalize about *how* effects apply yet — that's a second-instance problem, same reasoning as `Cell.hasCharge`.

Still not built: an upgrade shop with more than one item worth comparing, hub visuals that grow with purchases, offline passive generation. All deferred on purpose, same as before — build the next layer once the current one is actually felt to be missing something, not preemptively.

## Done: the world got a real scale, and a real belt (this session)

Prompted by: the map didn't actually resemble the spec's twin-star system at all — no home star, no far star, one asteroid 620px from spawn (about 1.3s of flight), a hub barely bigger than the ship. "Distance from home" couldn't mean anything because there was nothing to be distant *from*. Fixed as a deliberately scoped first pass — landmarks only, far star explicitly deferred, "inner starting area" only.

**Reference system:** loosely Sirius — a bright ordinary primary (home star, built) and a white-dwarf companion (far star, not built). Chosen over Alpha Centauri specifically for the asymmetry: the spec wants the two stars to feel qualitatively different (danger, resources, visual identity), and Sirius's mismatched pair does that for free instead of needing invented justification.

**Landmarks vs. content:** home star position, hub position, and the belt's own inner/outer radius are fixed landmarks (`HOME_STAR_POS`, `BELT_INNER_RADIUS`/`BELT_OUTER_RADIUS` in constants.ts). What's actually scattered *inside* the belt — the asteroids themselves — is procedural content generated at construction (`Engine.scatterBelt`), not permanent map geometry; it can grow or regenerate later without touching the landmarks. Risk/distance is measured from the **hub**, not the star (a deliberate simplification — the star exists for flavor/solar-power justification, the hub is what you actually navigate relative to).

**Scale:** ship stays the reference unit (unchanged). Hub 34px → 160px (a real station, not a big rock). Belt: 11,000–95,000px from the hub (~25s to the near edge, ~3.5min to the far edge at cruise speed, with no fast-travel yet — a deliberate choice, "Spacious" over "Compact," meaning real dead-air flying is expected until speed upgrades exist). Asteroids now come in three size classes (`ASTEROID_SIZE_CLASSES`) instead of one fixed template, scattered with a power-law bias toward the inner edge so the belt reads as denser near home with a sparser tail reaching the outer boundary, not uniformly random. Each asteroid also gets a randomized composition-weight archetype (`ASTEROID_ARCHETYPES` in asteroid.ts — ore-rich, crystal-rich, unstable-rich, balanced) so different rocks are actually worth visiting for different reasons.

**Consequence that had to be fixed in the same pass:** ping/vision were tuned for a world a few hundred px across (`PING_MAX_RADIUS` was 1600). Left alone, ping would have become useless the instant this landed — you can't find anything in a 95,000px belt with a sensor that reaches 1,600px. Rescaled `PING_MAX_RADIUS`/`PING_SPEED`/`VISION_RADIUS`/`SCAN_RANGE`/`SCAN_DATA_DISPLAY_RANGE` alongside the belt itself. Ship speed was deliberately *not* touched — that's the expected job of a future upgrade, not a stopgap here.

**The `asteroid` → `asteroids[]` refactor** touched most of `engine.ts`, but the shared-system work from earlier in the session paid for itself here: `resolveContact`/drift-group physics, `getContacts()` (radar), and the generic hover/highlight system needed **zero changes** — they were already written against "cells" and "drift groups," never against "the one asteroid." What actually needed generalizing: `findCellContact`/`nearestDrillableCell` (nested loop over every asteroid's cells — cell counts are small enough that no per-asteroid distance pre-filter is needed, and one would be unsafe anyway since drifted cells can be far from their origin asteroid's fixed center), `recomputeDriftGroups` (merges every asteroid's cells + neighbor graph into one flat pool — safe because cell ids are globally unique across asteroids already), and scanning (now "nearest asteroid," exposed as `Engine.nearestAsteroid` for the HUD, rather than a single global flag). One genuine correctness improvement fell out of this for free: laser/extraction "eject outward" direction used to reference the origin asteroid's fixed spawn center (`asteroid.center`), which goes stale once a piece has drifted far from home — now uses the piece's own current drift-group center of mass (`Engine.outwardDirection`), which is both more correct and doesn't need per-cell "which asteroid did this come from" bookkeeping.

**New: procedural infinite starfield** (`starfield.ts`). The old decorative background was a fixed list of 260 stars in a 1600px box — at belt scale, most of a real expedition would fly through empty black. Replaced with a deterministic hash-based generator: stars are computed on the fly for whatever world-space region is currently visible (a few dozen per frame, regardless of how far from the origin the camera has traveled), with the same grid cell always producing the same stars so revisiting an area looks stable rather than randomly regenerated.

**Deliberately not built yet:** the far star and its own zone (explicitly deferred — "get the inner starting area done first"), any distance-based risk scaling (the geometry now supports it, nothing reads it yet), belt content regeneration/depletion over time, real orbital motion for the home star. All flagged as the next layer, not built preemptively.

Note: `PING_MAX_RADIUS` was subsequently pulled back down from the 25,000 chosen here to 3,500 — reach-across-the-belt turned out to be the wrong instinct; ping is meant to reveal "just past the horizon," not let you skip exploring. Longer range is meant to be a future upgrade, not the baseline.

## Done: localized gravity — a well around the home star, not ambient physics (this session)

Requested as "better environmental interaction... gravitational pull, orbiting." The spec does mention this (§10, Phase 4 "Orbital/Phase-Cycle Timing") but its own wording explicitly rules out what was initially described: *"a slow, cyclical (not fully simulated) shift in lighting/danger... not full orbital physics... not required to validate whether the core loop is fun."* Real ambient gravity everywhere would also have reopened ship movement feel — the first thing validated in the whole build order, deliberately locked down early — and would need every asteroid in the belt to have a real orbit or it'd all eventually crash into the star, which is exactly the "full orbital physics" the spec says not to build. Landed on a middle ground instead: a **localized** well around specific big bodies, not a force that changes how normal cruising feels everywhere else.

**`gravity.ts`** (new, small, data-first — same shape as `explosion.ts`/`body.ts`): `GravitySource` is `{ kind, pos, radius, pullRadius, pullStrength, lethal }`, and `gravityAccel(source, pos)` is a pure function. Falloff is linear (same shape `Explosion` already uses), strongest right at the physical surface and zero at `pullRadius` — deliberately not inverse-square, both to avoid a singularity and to keep it tunable without a physics background.

**`Engine.gravitySources: GravitySource[]`** — today just the home star, but a list from the start specifically so a planet, moon, or the far star later is one more entry, not a new mechanic. `applyGravityTo(mover, dt)` takes anything with `{pos, vel}` (works today for the ship and every chunk via the same call), and runs inside the existing substep loops from the tunneling fix — a fast-orbiting ship skipping past a lethal surface in one big step is the exact same class of bug substepping already exists to prevent. Rocks/asteroids are deliberately exempt (would need real orbits, see above).

**Lethal, on purpose — "shouldn't just pull you in."** Crossing into a lethal source's physical `radius` (not just the pull radius) sets the ship's hull to 0, reusing the existing destroy/respawn path (`Engine.handleShipDestroyed`, now takes an optional `cause` so death-by-gravity gets its own message instead of the generic one — a first real consumer of the `DamageSource` tagging built earlier this session for exactly this kind of thing). A chunk that drifts in is just despawned, no ship consequence. The well itself (home star: pull radius 4000px, comfortably inside the hub's 6000px distance so docking is never affected) is tuned to be escapable with full thrust but not with none — drift in unpowered and the pull alone carries you the rest of the way in.

**Rendering is generic over the whole list** (`Renderer.renderGravitySources`) — a dashed warning ring at `pullRadius`, a tighter red ring at the lethal surface. A future planet gets both automatically just by existing in `gravitySources`, with zero renderer changes; only its own distinct *appearance* (a "what does a planet look like" concern, separate from `renderHomeStar`) would need new code.

**Follow-up in the same session: radiant heat.** The instant-death surface stayed unconditional (still exactly "touch it, you die"), but a second, independent hazard was added for the approach: `GravitySource.heatRadius`/`heatIntensity` (both optional — a source that doesn't set them radiates no heat at all, so a future cold body just omits them rather than needing a sentinel value), and `radiantHeatExposure(source, pos)`, same linear-falloff convention as `gravityAccel`. Deliberately kept as its own axis rather than folded into the pull itself — a future body could have gravity without heat, or heat without much pull.

**Rebalanced almost immediately — "too unforgiving."** The first cut applied heat as direct hull damage the instant you were in range, with no warning. Replaced with a two-stage model: exposure now raises `Ship.temperature` (0-100, decays passively when not exposed, same shape as `signature`) with **zero hull damage** below `TEMPERATURE_DAMAGE_THRESHOLD` (65) — a pure warning window. Only past that threshold does it start costing hull, ramping toward `TEMPERATURE_MAX_DAMAGE_PER_SEC` at full overheat (100). This is what makes `DamageSource`'s `"heat"` variant the first one actually *consumed* beyond being tagged — a burn death gets its own message ("BURNED UP") the instant it happens, distinct from touching the surface directly ("PULLED INTO THE STAR"), via `handleShipDestroyed`'s `cause` parameter. A one-time "HULL TEMPERATURE CRITICAL" message fires exactly on the threshold crossing (tracked by comparing temperature before/after that frame's update, not spammed every tick). Rendering: a HUD temperature bar (only shown once it's actually rising), plus the existing warm dashed ring at `heatRadius` in the field.

Caught and fixed right after: heat damage didn't work *at all*. `Ship.updateTemperature`'s first version computed `delta = exposure * RISE - DECAY` unconditionally, every tick — including while exposed. At the star's surface (the closest survivable distance) max exposure only produced ~11.6%/sec of rise, always less than the flat 15%/sec decay, so temperature could never mathematically climb regardless of how close you got. Fixed by only decaying when exposure is exactly zero (fully outside the heat radius) rather than racing rise against decay at every exposure level — simpler to reason about and avoids needing the two rates cross-tuned against each other.

**Also this session: the star joined the contact/radar system, and rocks got real gravity.**
- The star is now a radar `Contact` (new `ContactKind: "star"`), fed into `getContacts()` alongside rock — so it has to be found the normal way (ping sweep or passive vision range) and goes stale/forgotten like anything else on radar. First cut wrongly gave it the hub's always-known treatment; corrected — a fixed landmark isn't automatically something you already know about, only home base is. `GravitySource` gained a stable `id` field specifically so any future source can register itself as a contact the same way (`Engine.gravitySourceContact`, generic over the whole list, not hardcoded to the star).
- The belt got denser and more varied: `BELT_ASTEROID_COUNT` up to 16 full multi-cell bodies, plus a new separate population of 20 small single-cell boulders (`BELT_TINY_ROCK_COUNT`/`TINY_ROCK_RADIUS`) using the same scatter/bias logic. Made possible by a small `Asteroid`/`AsteroidOptions.seedCount` override — a tiny rock forces exactly 1 Voronoi cell (the whole body) instead of going through the normal size-based cell-count formula, which would still subdivide a small rock into several silly-tiny cells.
- Most belt bodies still start completely at rest, but a fraction (`BELT_DRIFT_CHANCE`, 35%) now spawn with a small random initial velocity (`BELT_DRIFT_SPEED`, 5-20px/s) — `Asteroid.initialVelocity`, picked up once by `recomputeDriftGroups` on the very first frame a body's drift group is created, then irrelevant (the group's own `vel` takes over).
- **Rocks now feel gravity too** — the earlier "localized well, not full orbital physics" design deliberately exempted them; that exemption was explicitly lifted this session. `applyGravityTo` (already shared by ship/chunks) now also runs on every drift group's center of mass each frame in `updateDriftGroups`, before the at-rest early-continue (so a stationary body near a well actually starts moving). This still isn't real orbital mechanics — no orbit is computed or maintained, bodies just feel the same local pull as everything else — and it stays safe in practice because the belt's inner edge (11,000px from the hub) sits entirely outside the home star's pull radius (max reach ~10,000px from the hub), so nothing drifts into a well by accident; only a body actively drifting inward over a long time would ever reach one.
- **Contact with a lethal body destroys the rock outright, with nothing dropped** — `destroyGroupsInLethalContact` marks every cell in a doomed group `fractured` (same flag normal extraction uses) but skips `spawnChunk` entirely, so no materials, no chunk, just gone. `recomputeDriftGroups` picks up the now-fully-fractured group as gone on its own next frame, the same mechanism that already handles any other cause of cells being fractured.

## Done: heat rebalance round 2, death screen, guaranteed near-hub rocks (this session)

**Bug: temperature wasn't reset on respawn.** `handleShipDestroyed` reset hull, cargo, position, and charges, but not `Ship.temperature` — dying near the star and respawning left you still carrying whatever temperature you'd built up, decaying passively from there instead of starting clean like every other post-respawn stat. One-line fix: `this.ship.temperature = 0;` added alongside the other resets.

**Rebalance: a deliberate full-speed dive into the star should hurt, not do nothing then suddenly kill you.** Round 1's numbers (`TEMPERATURE_RISE_PER_HEAT_UNIT = 0.4`, `TEMPERATURE_DAMAGE_THRESHOLD = 65`) were tuned only to fix the "never rises at all" bug, not actually balanced — worked through the math for a full-speed straight-line approach across the ~2900px heat zone at cruise speed (~473px/s) and found it would only reach roughly 65-90% temperature by the moment of lethal-surface contact, leaving little to no window for real hull damage before the instant-death collision (i.e. "nothing happens, then you're just dead," exactly what was reported). Raised `TEMPERATURE_RISE_PER_HEAT_UNIT` to 1.3 and lowered `TEMPERATURE_DAMAGE_THRESHOLD` to 50 — calculated to cost roughly 45 hull points over a full-speed direct approach, while still being fully survivable if the ship peels off before touching the surface. Not yet played live (no browser tool in this environment); flagging that these two numbers are the most likely to need a second pass once actually flown.

**Death screen.** Previously death only produced a transient HUD message, identical to any other status text and easy to miss. Added `Engine.deathScreen: { cause, cargoLost, timer } | null`, set by `handleShipDestroyed` (replacing its old `setMessage` call) and ticked down each frame in `update()` like the existing message timer, auto-clearing after `DEATH_SCREEN_DURATION` (3s) — deliberately non-blocking, doesn't pause or require a keypress, matching the "no hard fail state, respawn and keep going" feel the rest of death handling already has. `Renderer.renderDeathScreen` fades a dark overlay in/out and shows the cause text plus cargo lost (if any), reusing the same `cause` string `handleShipDestroyed` already produced for the old message (e.g. "PULLED INTO THE STAR", "BURNED UP").

**Guaranteed rocks near the hub.** The belt's inner edge sits 11,000px out, and scatter placement across the whole annulus is randomly biased but not guaranteed — early testing found long stretches near spawn with nothing nearby at all, undercutting the "always something to do nearby" feel. `spawnBeltBody` gained an options object (`{ seedCount?, distRange? }`, replacing the old positional `seedCount` argument) so the same placement/composition/drift logic can target a different distance band without duplicating it. `scatterBelt` now adds a third population — `NEAR_HUB_ROCK_COUNT` (10) single-cell boulders scattered across `NEAR_HUB_ROCK_RADIUS` (1500-7000px from the hub, well inside the main belt's inner edge) — guaranteed every run, independent of the wider belt's random bias.

**Ping range bumped alongside it.** `PING_MAX_RADIUS` 3500→5500, `PING_SPEED` 1400→2200, so a single ping comfortably covers the new near-hub rocks plus a bit of the belt's actual inner edge, rather than needing two pings back-to-back just to see what just got added nearby.

## Done: scan-scales-with-size, contact staleness overhaul, scale audit, unidentified contacts (this session)

**Scan duration now scales with the target.** `SCAN_HOLD_SECONDS` was a flat 1.6s regardless of whether the asteroid was a 20px boulder or a 420px giant. Added `scanSecondsForRadius` (asteroid.ts) — same sqrt-scaled shape as the existing `seedCountForRadius` (sublinear: a big rock takes noticeably longer, not proportionally longer), clamped between `SCAN_SECONDS_MIN`/`SCAN_SECONDS_MAX` (0.6s-3.2s) so nothing is instant or absurd.

**Contact staleness was way too generous, and had no spatial dimension at all.** Contacts were already forgotten after `CONTACT_FORGET_AFTER` and faded over `CONTACT_FADE_DURATION` — but at 45s/8s, a blip sat fully solid for 37 seconds before even starting to fade, reading as "stuck" rather than as a live sensor picture. Shrunk to 16s/6s. More importantly, added a second, independent reason to drop a contact: simple distance. `CONTACT_MAX_RANGE` (7000px, comfortably past `PING_MAX_RADIUS` so an edge-of-ping catch doesn't instantly vanish) — once the ship is farther than this from a contact's last-known position, it's dropped regardless of how recently it was refreshed. A ping is a local snapshot of what's nearby, not a permanent mark you can wander arbitrarily far from and still see. The hub stays the one exception to both rules, same as before.

**Scale audit.** Triggered by "an asteroid bigger than the sun" — investigated and found the actual bug wasn't the numbers (large asteroids capped at radius 420-500, the star's radius is 1100) but the star's *rendering*: `renderHomeStar` only drew a fully-opaque core out to `r*0.22`, with the rest of its radius spent on a gradient that faded to near-transparent by `0.4r`. A fully-opaque large asteroid could easily out-read that tiny solid core even though the star's true footprint dwarfed it. Fixed by widening the opaque core to `r*0.5` and pushing the gradient's bright stops outward, so the star now reads as a genuinely big body with a modest corona instead of a small dazzling point with a huge faint halo — a pure rendering fix, `HOME_STAR_RADIUS` itself (and therefore gravity/heat mechanics) is untouched.

While auditing further: `HUB_RADIUS` (160) was smaller than a medium asteroid's max (280) — directly contradicting its own doc comment ("bigger than most asteroids too"). Bumped to 220 (with `HUB_DOCK_RANGE` scaled alongside, 260→340, to preserve the same margin beyond the physical edge) so the hub clearly beats small/medium asteroids outright; a rare large asteroid can still out-mass it, which is fine — a big rock dwarfing a small station is a feature. Radar indicators for a not-yet-identified contact show a neutral, uninformative color and the label "Unidentified" instead of the kind-specific color/label (green hub, amber star, cyan rock) — that color coding was itself leaking identity before this fix, so it's now gated the same way the text label is. ## Done: the belt is a real ring around the star now, not a hub-wide annulus (this session)

**What was wrong.** "The belt" was `BELT_INNER_RADIUS`-`BELT_OUTER_RADIUS` (11,000-95,000px) around the *hub* — an 84,000px-wide annulus covering nearly the entire play area, with placement biased toward its own inner edge. Asked to spawn at "the edge of the belt" to look at it, and there was visibly nothing there: bodies are scattered uniformly in *angle* across the full circle, so even the theoretically-denser inner region has very low density at any single point you actually stand at — a huge annulus with ~50 bodies scattered across it doesn't read as "a belt," it reads as empty space with the occasional rock. Direct feedback: a belt should be a dense ring around the *star* (real solar-system logic, and consistent with the just-built star-anchored coordinate system), a deliberate landmark "to reach," clearly denser than its surroundings — and the space between the hub and the belt shouldn't be near-empty either, it should have a moderate, mostly-smaller-sized scatter of its own.

**Rebuilt as three distinct, deliberately-sized populations** (`Engine.scatterSystem`, replacing `scatterBelt`; `spawnSystemBody`, replacing `spawnBeltBody` — now takes an explicit `origin`, not always the hub):
- **The belt** — a real ring around `HOME_STAR_POS`: `BELT_INNER_RADIUS`/`BELT_OUTER_RADIUS` now 28,800/40,800 (4.8-6.8 AU, a 2 AU-wide band centered ~5.8 AU out). Count roughly doubled (16→24 main bodies, 20→24 tiny rocks) since the ring is now a much smaller area than the old annulus and needs more bodies to actually read as dense. `BELT_SIZE_POOL` skews its size-class draw toward medium/large — the belt is where the big, "worth traveling for" bodies live.
- **Normal space** — also star-anchored, filling `NORMAL_AREA_INNER_RADIUS` (7,200, just past the star's own hazard radius) to `NORMAL_AREA_OUTER_RADIUS` (= the belt's own inner edge, so the two bands meet with no gap and no overlap). Moderate counts (18 + 16), `NORMAL_AREA_SIZE_POOL` skewed hard toward small — something to actually mine on the way out, without competing with the belt as the destination.
- **Near-hub boulders** — unchanged, still hub-relative (`this.hub.pos`, not the star) — a small guaranteed population right outside the hub regardless of where the other two bands' randomness lands.

Dropped the old inner-edge power-law bias (`Math.random() ** 2`) for all three — that was specifically compensating for one huge span needing to taper gradually; three separate, deliberately-sized bands don't need it, so placement within each is now uniform across its own distance range.

`Renderer.renderBeltBoundary` now draws both the belt's inner *and* outer edge (previously just one ring) centered on `HOME_STAR_POS` instead of the hub.

## Done: three more mechanics fixes from live testing (this session)

**Scan sweep now follows a moving/drifted asteroid.** The scan-in-progress circle (and the "which cells are inside the sweep" test) read `Asteroid.center` — which was only ever the *original spawn point*, frozen forever. Fine for a body that's never moved, visibly wrong for one drifting under gravity (rocks have felt gravity since earlier this session) — the circle stayed put while the actual rock moved away from it. ## Done: real frame-rate regression from the belt redesign — fixed (this session)

Reported immediately after the belt redesign landed: "running terribly." Root cause: `resolveGroupCollisions` (rock-vs-rock, runs every frame unconditionally) swept every pair of drift groups — O(driftGroups²) — with **no group-level bounding check at all** before diving into `resolveGroupPair`'s full per-cell double loop. Two asteroids on opposite sides of the belt still paid the full per-cell-pair cost (including a `boundingRadius` call recomputed redundantly for the same cellB on every cellA iteration). This was always an inefficiency, just not one that mattered at ~46 bodies. The belt redesign roughly doubled total body count (~46→~92) and skewed size selection toward medium/large (more cells per body) — doubling the group count quadruples the O(n²) pair count, and the per-pair cost went up too, compounding into a genuine, user-visible slowdown rather than a theoretical one.

Fixed with the same cheap-reject pattern already used elsewhere this session (`findCellContact`, `cellAt`, `nearestDrillableCell`): `resolveGroupCollisions` now computes each group's center of mass and bounding radius once per frame (O(total cells), unavoidable and already cheap), then does an O(groups²) *bounding-circle-only* check — a couple of subtractions and one `distance` call — before ever calling `resolveGroupPair`. Only pairs that are actually close pay for the expensive per-cell work. `resolveGroupPair` itself also stopped recomputing `boundingRadius` for the same `cellB` on every `cellA` iteration — precomputed once per group-B cell instead.

Lesson for next time a spawn-count/size constant changes: the physics audit earlier this session found and fixed three other unbounded per-frame scans (`findCellContact`'s existing pattern, then `cellAt`/`nearestDrillableCell`), but this one — arguably the most expensive of the four, since it's O(n²) rather than O(n) — was missed because at the time nothing had changed the group count enough to expose it. Worth specifically re-checking every O(n²) loop (there's only this one) whenever body counts change meaningfully again.

## Done: a fuller perf sweep after the O(n^2) fix (this session)

Asked to check for anything else poorly optimized, following the `resolveGroupCollisions` fix above. Found and fixed four more, all the same family of bug (a hot path visiting every cell/asteroid in the world instead of only the ones that matter), roughly in order of impact:

- **`driftGroupOf` was O(total cells) per call** (`driftGroups.find(g => g.cells.includes(cell))`) — called from `exposedEdgeMask`, itself called once per nearby candidate cell inside the already-fixed `findCellContact`/`nearestDrillableCell`. Fixed with `cellGroupMap: Map<number, DriftGroup>`, rebuilt once per frame inside `recomputeDriftGroups` (which already visits every cell regardless), so lookup is now O(1).
- **`resolveGroupCollisions` and `getContacts` each independently recomputed every group's center of mass and bounding radius**, once per frame, redundantly. Consolidated into one `computeDriftGroupBounds()` call in `update()` (right after `updateDriftGroups`, before anything reads it) — cell positions are settled for the rest of the frame at that point, so both consumers can safely share the same pass instead of duplicating it. (Verified the one place that invalidates it mid-frame — `detonateCharges`'s own `recomputeDriftGroups()` call — happens *after* both consumers have already read it for this frame, so nothing goes stale before it's used.)
- **`renderAsteroids` drew every cell of every asteroid unconditionally**, every frame, regardless of whether any of it was anywhere near the camera — the system now spans a belt ring plus a normal-area band, both far bigger than one screen. Added a cheap bounding-circle-vs-viewport reject (generous 300px margin for outline noise) before descending into a given asteroid's cells at all.
- **Two renderer lookups searched every asteroid for "which one owns this cell"** (`engine.asteroids.find(a => a.cells.includes(hoveredCell))` for the hover tooltip, same pattern for the RCS target-scanned check) — O(total cells) each, once per frame apiece. Fixed by giving `Cell` a permanent `asteroid` back-reference, set once at creation (unlike drift-group membership, a cell's origin asteroid never changes), making both O(1).

Swept the rest of engine.ts/renderer.ts for the same pattern and found nothing else comparable: the remaining full-cell scans (`destroyGroupsInLethalContact`, `updateDriftGroups`'s rotate/translate pass, `recomputeDriftGroups`'s own BFS) are all *necessary* per-frame work — everything genuinely needs a look each frame, there's no "nearby subset" to filter to. The handful of other O(total cells) scans that remain (`detonateCharges`'s charge-collection, `nearestDriftGroup` for explosion targeting) are event-driven — they run once per player action (detonating charges), not every frame — so they're not worth the same treatment.

## Done: bug — a fully-mined asteroid kept showing on radar (this session)

Reported after the perf sweep above. Cause wasn't performance — `recomputeDriftGroups` already correctly drops a drift group the instant its last cell is fractured (no component survives a BFS with zero intact cells feeding it), so the body itself was genuinely gone right away. The bug was in `discoveredContacts`'s forget loop: it only ever checked *staleness* (age past `CONTACT_FORGET_AFTER`) or *distance* (past `CONTACT_MAX_RANGE`) — never "does this thing still exist at all." A destroyed body's last-known position just sat there decaying on the normal timer like any other memory going stale, instead of vanishing the moment it was actually gone — up to 16 seconds of a radar blip for something that no longer existed, longer if the ship stayed nearby (only the distance check would have caught it, and being nearby is exactly when that doesn't trigger).

Fixed by adding a third removal reason: `getContacts()` always lists *every* rock body currently alive anywhere in the world (not just nearby/discovered ones — proximity filtering is a separate, later step), so "this id isn't in this frame's `contacts`" reliably means destroyed, not just out of range. The forget loop now drops a `discoveredContacts` entry immediately if its id isn't in the current live contact set, same frame the underlying body disappears — no separate cleanup pass needed, since `recomputeDriftGroups` already runs unconditionally every frame before this check.

## Done: a real mineral/ore system, replacing the placeholder three (this session)

Spec'd first (twin-star-spec.md Section 17) then implemented — the request was explicitly "proper mineral, ore, element system... realistic ores, things that could logically be found in space... depth but not sim level," and an earlier attempt that just renamed the same three generic buckets was correctly rejected as not meeting that bar.

**Asteroid type is the new load-bearing concept.** `AsteroidType` (`asteroid.ts`) — `"S" | "C" | "M" | "Icy"`, real simplified spectral classification — determines which resources a body can even contain, not just flavor on top of a universal chance. `RESOURCE_WEIGHTS_BY_TYPE` maps each type to its own resource-weight profile (S-type: rock+iron-dominant; C-type: mostly rock, less metal; M-type: metal+platinum only, no filler; Icy: ice-dominant with real rock content, a "dirty snowball" like actual comets). Every type carries a small Radioactive Ore chance — "any type, always rare" from the spec, satisfied by baking a small slice into every table rather than a separate mechanic.

**Six resources replace the old ore/crystal/unstable**, each with a real-world basis (see spec Section 17 for the reasoning per resource): Chondrite Rock (bulk filler), Nickel-Iron (structural metal), Silicate Crystal (renamed from the old "Dense Crystal," same role), Platinum-Group Ore (new, M-type-exclusive, the actual reason real asteroid-mining proposals target metallic bodies), Water Ice (new, Icy-exclusive, deliberately rare this close to the star, not spendable on anything until the still-unbuilt Fuel/Power systems exist), and Radioactive Ore (renamed from "Hollow-Unstable" — recommended tool is charges for an explicit *safety* reason now, place-and-retreat, not a hardness one).

**Zone-based type distribution, tying directly into the belt redesign.** `NORMAL_AREA_TYPE_WEIGHTS` (85% S-type — safe, unremarkable, on purpose) vs `BELT_TYPE_WEIGHTS` (45% S, real C/M/Icy presence) — engine.ts's `spawnSystemBody` now picks a type per body from whichever zone it's spawning into, before the `Asteroid` constructor picks that specific body's resource mix from the type's table. This is the material reason the belt is worth reaching, not just visual density: Platinum-Group Ore, Water Ice, and Radioactive Ore only become realistically findable there.

**New shared utility: `weightedPick` (`random.ts`).** The old composition picker was hand-specialized to exactly three fields (`ore`/`crystal`/`unstable`, literally hardcoded by name). With composition-within-type, type-within-zone, and (already existing) size-class-within-zone all needing the same "weighted category pick" shape, this crossed the rule-of-three threshold for a real shared helper — a generic `Record<key, weight>` + roll → key function, reused by `asteroid.ts`'s composition picker and `engine.ts`'s type picker. (The pre-existing `BELT_SIZE_POOL`/`NORMAL_AREA_SIZE_POOL` index-repetition arrays were left as-is — not in scope for this change, though they're a candidate to migrate onto `weightedPick` too for consistency, next time that code is touched.)

**`CargoHold` (materials/cargo/upgrade-cost, one shared shape) stopped being hand-written per field.** With 3 fields it was fine for `emptyCargo`, `cargoUsed`, `canAfford`, and the upgrade-purchase debit to each name `ore`/`crystal`/`unstable` explicitly; with 6 it was a maintenance trap (four separate places that would each silently miss new resources). Added `COMPOSITIONS` (`asteroid.ts`, `Object.keys(COMPOSITION_INFO)`, so the list itself can never drift out of sync with the info table) and rewrote all four call sites plus two more hardcoded UI lists (`HubOverlay.tsx`'s material list and cost display, `renderer.ts`'s SCAN DATA breakdown) to iterate it generically instead.

**Scan now reveals asteroid type alongside composition** — the SCAN DATA panel header shows e.g. "S-type (Silicaceous)", and the composition breakdown below it only lists resources that body's type could actually contain (showing "Platinum-Group Ore: 0 intact" on a body that could never have had any would read as a missed find rather than an impossible one).

**Not decided/built yet, per the spec's own open items:** exact hardness/value/tool numbers are a first pass, not tuned by play; whether asteroid type should be visible pre-scan (e.g. color-coded) instead of scan-gated like composition; whether Water Ice gets a heat-proximity-based degradation mechanic. Also unbuilt, unchanged from before: the Fuel/Power systems Water Ice is earmarked for.

## Done: cargo capacity and material weight are now a real metric, not an abstract count (this session)

Requested as "a proper metric for measuring capacity and weight of materials/rocks." Cargo capacity was 16 abstract "units" with every resource contributing 1:1 regardless of what it actually was — no distinction between a lump of rock and the same amount of platinum.

**Each resource got a real density** (`CompositionInfo.density`, g/cm³) — actual approximate figures for the real material each resource is grounded in (ordinary chondrite 3.3, iron-nickel meteorite 7.9, olivine 3.5, water ice 0.92 — deliberately *less* dense than rock, same as reality — platinum-group and radioactive ore priced as enriched host-rock ore rather than the pure metal/mineral's real, much higher density, since "Platinum-Group *Ore*" isn't pure platinum).

**Cargo capacity is now `CARGO_CAPACITY_KG` (750kg)**, a real mass limit, not a slot count. `weightKgFor(composition, amount)` (asteroid.ts) converts a resource's existing size-scaled `chunkValue` amount into an actual kg figure via `amount * density * MATERIAL_WEIGHT_SCALE` (a tuned conversion constant, not a literal m³ volume calculation — grounded relative densities, not full unit-precise 3D volume tracking, matching the same "depth, not sim-level" rule Section 17's minerals were built under). `Ship.cargoUsed` sums real weight across all held resources; `addCargo` now computes how much *amount* of a resource fits given remaining *weight* room (a dense material like Nickel-Iron fills the hold faster per unit than light Water Ice, a genuine, realistic tradeoff that didn't exist before).

**Extended to `Chunk.mass` too, for physical coherence** — a chunk's actual collision-physics mass (used throughout the existing rigid-body system for momentum/knockback) now scales by the same relative density (relative to rock, the baseline `ROCK_MASS_PER_AREA` was implicitly tuned against), so a Nickel-Iron chunk is genuinely harder to shove around than a Water Ice chunk of the same size — the same weight distinction shows up in how it physically behaves, not just in the cargo number. **Deliberately not extended** to the asteroid/drift-group mass system (`groupMass`/`groupCenterOfMass`/`groupInertia`, used throughout the core rock-vs-rock and ship-vs-rock collision resolvers) — that's spread across many more call sites in code that was specifically audited and confirmed physically consistent earlier this session, and making density affect *asteroid* physics (not just loose chunk debris) is a materially bigger, riskier change than what was actually asked for. Flagged here in case that fuller version turns out to be wanted later.

**UI updated to show the real unit everywhere it appears**: the HUD cargo bar reads `{used}/{capacity}kg`, the hub overlay's capacity readout and upgrade descriptions/bonus are in kg, and the chunk hover tooltip — added last session for cargo-fit purposes — now shows actual kg instead of the old abstract "value" number, with the same fits/partial/full color-coded states as before just measured correctly now.

## Done: one density-aware mass system, not two disconnected ones (this session)

Direct follow-up to the cargo-weight work — flagged there as a deliberately descoped decision, and asked for by name: "audit the asteroid stuff to the new system, I don't want multiple metric systems going around." Last session gave `Chunk.mass` real per-resource density, but left the asteroid/drift-group mass system (`groupMass`, `groupCenterOfMass`, `groupInertia`, and this session's own `recomputeDriftGroups` center-of-mass weighting) using the old flat `ROCK_MASS_PER_AREA` regardless of composition — meaning a Nickel-Iron cell and a Water Ice cell of the same area had identical physics mass right up until the instant of extraction, when the chunk suddenly became correctly denser or lighter. Two disconnected mass conventions for the same material, exactly the kind of thing this session already fixed once for cargo/chunk value before the mineral system existed.

Fixed with a single shared pair of functions in asteroid.ts — `massPerAreaFor(composition)` (`ROCK_MASS_PER_AREA` scaled by that resource's real density relative to rock's) and `massForArea(composition, area)` — now the *only* place area becomes physics mass anywhere in the game. `Chunk.mass` was rewired to call it instead of keeping its own inline copy of the same formula. `groupMass`/`groupCenterOfMass` now sum per-cell mass using each cell's own composition (a Voronoi tessellation isn't single-material — a group can and does mix rock, metal, crystal, etc.), rather than treating every cell in a body as equally dense. `groupInertia` was split into `massPerArea`/`m`/`ownInertia` per cell for the same reason — the parallel-axis-theorem physics itself is untouched, only the per-cell mass feeding it is now composition-aware. The `recomputeDriftGroups` live-center weighting (added earlier this session for the scan-follows-moving-asteroid fix) got the same treatment, so a dense chunk within an asteroid now genuinely pulls that asteroid's own "current position" more than an equally-sized light one, consistent with everything else.

Net effect: an M-type asteroid (mostly Nickel-Iron/Platinum-Group Ore, both dense) now has real, correctly heavier collision mass than an equally-sized Icy body (mostly Water Ice, much less dense) — this was never true before and nothing about it was previously felt or tested, so it's worth a firsthand look once flown, same caveat as always about this environment having no browser to visually confirm gameplay feel.

## Known tradeoff: world-space vs. local-space transforms for drift groups

Rock pieces (`DriftGroup`s) that break loose from the asteroid can translate *and* rotate (real rigid-body physics — see `physics.ts`). The way that's implemented: each `Cell.polygon` is stored in absolute world-space coordinates, and every frame the group's rotation/translation delta is applied directly to those vertices in place (`rotateAround` + translate, mutating the array).

The textbook-correct alternative is to store each cell's geometry in **local space** relative to the group's center of mass, plus a single `(position, angle)` transform, and derive the world-space polygon on demand. That avoids any possibility of floating-point drift accumulating over many small incremental rotations, and is the standard approach in general-purpose physics engines.

I went with the world-space/incremental approach instead, deliberately:

- Cell geometry is *read* far more often than it's *written* — targeting, collision queries (`cellAt`, `closestBoundaryPoint`), and rendering all run multiple times per frame, while a group's transform is only updated once per frame. Storing world-space directly means every read is free; local-space-plus-transform would mean recomputing the world polygon on every single query instead.
- Floating-point error from repeated small rotations accumulates *extremely* slowly at double precision — far slower than any realistic play session would ever expose. This is not a tradeoff that matters at "browser game, minutes-to-hours session" scale.

This is the one place in the physics code where I traded theoretical correctness for practical performance. If a future need ever requires long-running deterministic replay, exact rewind, or multiplayer state reconciliation, this is the first place that assumption would need revisiting.
